<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API | Untrusted HTTP requests from web clients (browser, API consumers) |
| API → Database | Trusted internal queries (Kysely parameterized queries prevent injection) |
| API → External Services | Third-party integrations (Zulip, GitHub, etc.) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-01 | Tampering | Request body validation (all POST/PUT endpoints) | mitigate | Zod schemas in ts-rest contracts validate request body structure and types at API boundary before data reaches handlers |
| T-29-02 | Tampering | Response data leakage | mitigate | responseValidation: true ensures handlers don't return extra fields not defined in contract, preventing accidental data exposure |
| T-29-03 | Spoofing | Type confusion attacks | mitigate | Runtime Zod validation prevents attackers from bypassing type checks via malformed input (e.g., sending string where number expected) |
| T-29-04 | Information Disclosure | Error messages revealing internal state | mitigate | responseValidationErrorHandler returns generic error message, logs detailed error to server-side console only |
| T-29-05 | Denial of Service | Complex nested objects causing validation overhead | accept | Zod validation is fast; DoS protection via existing rate limiting middleware (not in scope for this phase) |
| T-29-06 | Tampering | SQL injection via query params | mitigate | Kysely parameterized queries already prevent this; contract validation adds defense-in-depth by type-checking params before use |

**Security-specific notes:**
- responseValidation: true ensures contract compliance even if handlers are buggy
- Zod schemas strip unknown fields by default (passthrough: false is default)
- Type inference prevents "any" types from bypassing validation at compile time
</threat_model>
