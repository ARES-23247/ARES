<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API | Untrusted HTTP requests from web clients (browser, API consumers) |
| API → Database | Trusted internal queries (Kysely parameterized queries prevent injection) |
| API → External Services | Third-party integrations (Zulip, GitHub, etc.) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-04 | Spoofing | Type confusion attacks | mitigate | Runtime Zod validation prevents attackers from bypassing type checks via malformed input (e.g., sending string where number expected) |
| T-29-05 | Denial of Service | Complex nested objects causing validation overhead | accept | Zod validation is fast; DoS protection via existing rate limiting middleware (not in scope for this phase) |
| T-29-06 | Tampering | SQL injection via query params | mitigate | Kysely parameterized queries already prevent this; contract validation adds defense-in-depth by type-checking params before use |

**Security-specific notes:**
- responseValidation: true ensures contract compliance even if handlers are buggy
- Zod schemas strip unknown fields by default (passthrough: false is default)
- Type inference prevents "any" types from bypassing validation at compile time
</threat_model>
