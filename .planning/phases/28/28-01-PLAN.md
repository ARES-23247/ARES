<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client request → Handler input | Untrusted external data enters system via HandlerInput |
| Handler → D1 Database queries | Typed data flows into database operations |
| Handler → Third-party APIs (Gcal, Zulip) | Data sent to external services |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-01 | I | HandlerInput types | accept | Type-only changes — no new validation. Zod validation comes in Phase 29 |
| T-28-02 | T | D1Row types | mitigate | Types prevent accidental field access but runtime validation (Zod) required for complete safety |
| T-28-03 | I | socialConfig as any removal | mitigate | Proper typing prevents incorrect config access, reduces risk of sending secrets to wrong service |
| T-28-04 | D | Response body types | accept | Type-only changes improve DX but don't change data disclosure |

## Security Notes

- Removing `as any` casts prevents accidental field access and type confusion bugs
- Proper typing of socialConfig prevents sending service account credentials to wrong endpoints
- Type-safe handler signatures prevent parameter swapping attacks (compile-time catch)
- Runtime validation (Zod) will be added in Phase 29 to complete the security boundary
</threat_model>
